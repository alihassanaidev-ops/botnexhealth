# CDK + ECS Deployment Target

## Decision

The long-term deployment target for this repo should be AWS CDK + ECS/Fargate, not Kubernetes and not continued investment in Copilot.

## Why this fits this app

The system shape is straightforward:

- one public HTTP API
- one private background worker
- one PostgreSQL database
- one Redis broker/session store
- one static frontend

That is exactly the kind of workload ECS/Fargate handles well without introducing Kubernetes operational overhead.

## What the new CDK target provisions

The `infra/` app provisions:

- ECS cluster
- public API load balancer + WAF
- Fargate API service
- Fargate worker service
- one-off migration task definition
- RDS PostgreSQL
- ElastiCache Redis with TLS
- recordings S3 bucket
- frontend S3 bucket + CloudFront distribution
- generated `JWT_SECRET` and `ENCRYPTION_KEY`

## What stays explicit

Some secrets should stay explicit and environment-owned rather than baked into infra code:

- `NEXHEALTH_API_KEY`
- `RETELL_API_SECRET`
- `RESEND_*`
- `TWILLIO_*`

The database credentials are generated by the stack and injected directly into the ECS tasks.

## Operational sequence

1. Deploy infra with CDK.
2. Update external secrets.
3. Run the migration task.
4. Deploy or publish the frontend.
5. Cut traffic over only after API and auth flows pass on staging.

## Runtime fixes bundled with this change

Two runtime issues were fixed alongside the new deployment target:

- Redis URLs are normalized for TLS so `rediss://` works cleanly with Celery and `redis-py`.
- S3 access now relies on IAM task roles instead of static AWS access keys in app config.
